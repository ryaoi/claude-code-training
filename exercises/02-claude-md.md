# 演習2: CLAUDE.md（メモリ）

## 目的

CLAUDE.md を**書く前と後で**、Claude の振る舞い（命名規則・色の扱い・既存コードへの追従度）が
どう変わるかを実体験する。

## 前提

- `sample-project/index.html` には Button / Card セクションがあり、Tag セクションは「未実装」と書かれている
- `css/components.css` は **BEM 命名**、色は `var(--color-*)` トークン参照、で書かれている

---

## 解説

### CLAUDE.md とは

**プロジェクトや自分の前提をファイルに書いておき、Claude Code 起動時に自動で読ませる仕組み**。
毎回の会話で同じ説明を繰り返さなくてよくなる。

```
<project>/CLAUDE.md   ← ここに書く
       ↓ 起動時に自動ロード
Claude が「このプロジェクトでは BEM を使う」などを最初から知っている状態で始まる
```

### メリット

- 「うちは BEM で書いて」「色はトークン使って」を**毎回説明しなくていい**
- チームで共有できる（git に含める） → 誰が使っても同じ Claude になる
- 規約の**単一情報源**になる（README とばらけない）

### 置き場所（覚えるのは3つだけ）

| 場所 | スコープ | 用途 | git に含める？ |
|------|---------|------|--------------|
| `<project>/CLAUDE.md` | プロジェクト | **チーム共有の規約**（今日メイン） | 含める |
| `~/.claude/CLAUDE.md` | 全プロジェクト | 個人の癖・好み（日本語で返して等） | （個人ファイル） |
| `<project>/CLAUDE.local.md` | プロジェクト | 自分だけのメモ（未完成の注意書き等） | 含めない（`.gitignore` に入れる） |

> **応用**: サブディレクトリにも置けます（例: `src/api/CLAUDE.md` は `src/api/` 配下でだけ効く）。
> 組織全体の設定ファイルもありますが、今日は上の3つで十分。

### 読み込み順（優先度）

```
低  ユーザー (~/.claude/CLAUDE.md)
 ↓
    プロジェクト (<project>/CLAUDE.md)
 ↓
高  ローカル (<project>/CLAUDE.local.md)
```

**下（プロジェクト）が上（ユーザー）を上書き**する。
「個人では英語で返してほしいけど、このプロジェクトだけは日本語」みたいな切り分けができる。

### 中身に書くもの / 書かないもの

**書く**:
- 技術スタック（何の言語・何のフレームワーク）
- よく使うコマンド（`npm test`、`open index.html` など）
- 命名規約・コーディング規約（**BEM 使う、トークン使う** など）
- ディレクトリ構成と各フォルダの役割

**書かない**:
- 秘密情報（API キー、パスワード、個人情報）
- 一般論（「きれいに書いて」「良いコードを」など、Claude が元から知ってる抽象論）
- 重複する内容（他ファイルに書いてあるなら `@<path>` で参照）

### 便利機能4つ

#### ① `@<ファイルパス>` で他ファイルを取り込み

```markdown
## コーディング規約

詳細は @docs/style-guide.md を参照してください。
```

→ 起動時に `docs/style-guide.md` の内容も一緒に読まれる。
規約が長い時は別ファイルに切り出して参照できる。

#### ② `/init` で雛形を自動生成（演習1の組み込みコマンド）

```
/init
```

→ Claude がプロジェクト全体を眺めて、`CLAUDE.md` の雛形を作ってくれる。
ゼロから書くのが面倒なら、まず `/init` してから手で整える、が定番。

#### ③ `/memory` でエディタを開く

```
/memory
```

→ どのメモリファイルを編集するか聞かれ、デフォルトエディタで開く。

#### ④ 自然文で「これ覚えて」と頼む

会話中に普通の日本語で:

```
これは CLAUDE.md に追加して: BEM 命名を徹底する。ハイフン1個区切りは使わない
```

→ Claude が `CLAUDE.md` に追記してくれる。

> 「`add this to CLAUDE.md`」と明示的に頼むと CLAUDE.md に入る。
> 単に「remember this」「これ覚えて」だと **auto memory**（Claude 自動学習領域、`~/.claude/projects/<project>/memory/`）に保存される。違いを意識しておくとよい。

### 書き方のコツ

- **具体的 > 抽象的**
  - NG: 「きれいに書いて」「ベストプラクティスで」
  - OK: 「BEM 命名 (`.block__element--modifier`)、色は `var(--color-*)` 参照」
- **長すぎると効果が落ちる**: 公式推奨は **200行以下**。長すぎるとコンテキスト消費が増えて遵守率も下がる
- **古くなったら更新する**: 書きっぱなしで実情と違うと逆効果
- **Before/After のコード例**を入れると伝わりやすい

## ハンズオン手順

### Step 1: CLAUDE.md なしで指示を出す

何も用意せずに、Claude にこう依頼（**色を指定するのがポイント** — 既存トークンに存在しない赤・黄・緑を要求する）:

```
index.html の「Tag (未実装)」セクションに、Tag コンポーネントを追加して。
Primary（赤）/ Secondary（黄）/ Default（緑）の3種類が並ぶようにして。
```

**観察ポイント**:
- 色は何で書かれる？（`#ef4444` などの**ハードコード直書き**？ それとも `tokens.css` に新トークンを追加してから参照？）
- `tokens.css` に追加された場合、命名はどんな規則？（`--color-tag-primary` / `--color-tag-primary-bg` / `--tag-red`… 試行ごとにブレる）
- Default 用の modifier (`.tag--default`) は作られる？ それともベース `.tag` で代用される？
- 「未実装」を示す `<small>` タグや `.section__note` 段落は削除される？ 残る？

> **補足**: BEM 命名 (`.tag--primary`) や `components.css` 末尾への追加は、既存の Button / Card のコードを Claude が読んで真似てくれるので、ほぼ確実に守られます。**CLAUDE.md なしで差が出るのは、既存コードから読み取れない判断**（新トークンの命名規則、未実装マーカーを消すか、衝突する色指定にどう対応するか）です。

> **試行ごとに結果がばらつく**のもこの演習の重要なポイント。隣の人と見比べると「自分はハードコード、隣はトークン追加」みたいな差が出やすい。**「規約を毎回口頭で言うより、文書化した方が安定する」**ことを体感する材料になります。

確認後、変更を全部戻す。git に不慣れなら、Claude に頼むのが一番早い:

```
さっき追加した Tag コンポーネント関連の変更を全部元に戻して、index.html と css/ を最初の状態にして
```

または、追加された箇所を手動で消してもよい。

### Step 2: CLAUDE.md を作成する

`sample-project/CLAUDE.md` を作成し、以下を貼り付ける:

```markdown
# プロジェクト概要

Claude Code 研修用の HTML/CSS コンポーネント集。
ビルドツールなし、ブラウザで `index.html` を直接開いて確認する。

## ディレクトリ構成

- `index.html` デモページ
- `css/reset.css` 最小リセット
- `css/tokens.css` デザイントークン（色・余白・角丸・フォント）
- `css/components.css` コンポーネント実装

## CSS 規約

### 命名: BEM 厳守
- Block: `.card`
- Element: `.card__title`（ダブルアンダースコア）
- Modifier: `.card--featured` / `.btn--primary`（ダブルハイフン）
- キャメルケースやハイフン1個の命名（`.cardTitle` / `.card-title`）は使わない

### 色・余白・角丸: トークン参照のみ
- 色は **必ず `var(--color-*)`** を使う。`#2563eb` 等のハードコードは禁止
- 余白も `var(--space-*)`、角丸も `var(--radius-*)` を使う
- 必要なトークンが無ければ、**先に `tokens.css` に追加してから**コンポーネントで使う
- 新トークンの命名は `--color-<コンポーネント名>-<バリアント名>`（例: `--color-tag-primary`）

### 配置
- 新しいコンポーネントの CSS は `css/components.css` の末尾に追記
- 区切りコメント `/* ====== Tag ====== */` を入れて他コンポーネントと視覚的に分ける
- 新しい CSS ファイルは作らない（今回はファイル数を増やさない方針）

### HTML
- 既存セクションの構造（`<section class="section">` + `.section__title` + `.section__body`）に合わせる
- 「未実装」を示す `<small>` タグや `.section__note` 段落は、実装完了時に削除する
- 日本語の文言を使ってよい

## 確認方法

ブラウザで `index.html` をリロードして、見た目で確認する。
スクリーンショットを取る必要はない。
```

> **ちなみに**: ゼロから書く代わりに、`/init` と打てば Claude がプロジェクトを分析して雛形を作ってくれます。今日は手で書きますが、実プロジェクトでは `/init` → 手直し、が楽。

### Step 3: 同じ指示を再度出す

Claude Code を `/clear` で履歴をリセットしてから、もう一度（Step 1 と完全に同じ指示）:

```
index.html の「Tag (未実装)」セクションに、Tag コンポーネントを追加して。
Primary（赤）/ Secondary（黄）/ Default（緑）の3種類が並ぶようにして。
```

**観察ポイント**（Step 1 とどう変わったか）:
- ハードコードが消え、**全色が `tokens.css` への追加 → `var(--color-tag-*)` 参照**に揃ったか？
- 新トークンの命名が `--color-tag-<バリアント>` パターンに統一されたか？（試行ごとのブレが消える）
- `<small>(未実装)</small>` と `.section__note` 段落が**確実に削除**されたか？
- `.tag--default` modifier が作られているか？

> Step 1 で試行ごとにブレていた部分が、CLAUDE.md ありだと **3回やっても同じ形**に揃うのが体感ポイント。

### Step 4: ブラウザで確認

`index.html` をリロードし、Tag セクションに3つのバッジが表示されることを確認。

### Step 5: 自然文で CLAUDE.md に追記する（1分・余裕があれば）

会話プロンプトで普通の日本語で:

```
これを CLAUDE.md に追加して: Tag は小さめなので font-size は var(--font-size-sm) を使う
```

→ Claude が `CLAUDE.md` を読んで、適切な場所に追記してくれる。
**思いついた規約をその場で保存できる**のが便利。

> 「`/memory`」と打てば編集画面も開けます（手で直接編集したい時用）。

---

## 期待される結果

- CLAUDE.md なし → ありで、**色の扱い（ハードコード vs トークン化）と新トークン命名のブレが揃う**
- Step 1 では試行ごとに結果がばらつき、Step 3 では3回やっても同じ形になる
- 「規約を都度言わなくていい」「3人でやれば3人とも同じ実装になる」体験ができる
- 規約はチームで共有資産になる、と実感できる

## CLAUDE.md が効くところ・効かないところ

| 観点 | CLAUDE.md なしでも守られる？ | 理由 |
|------|--------------------------|------|
| BEM 命名 (`.tag--primary`) | **守られる** | 既存の `btn--primary` `card__title` を読んで真似るため |
| `components.css` 末尾に追加 | **守られる** | 既存の追加パターンを読み取れるため |
| 既存トークンへの参照 | **守られる** | `tokens.css` を見れば `var(--color-*)` の存在に気づくため |
| **新トークンの命名規則** | **ブレる** | 既存コードに前例がないので推測になる |
| **既存にない色を要求された時の対応** | **ブレる**（ハードコード or トークン追加） | 同上 |
| **「未実装」マーカーの削除判断** | **ブレる** | 明示されないと残すかも |

→ **CLAUDE.md の真価は「既存コードから読み取れない判断」と「規約と衝突する指示が来た時の挙動」を揃えること**。
コードに痕跡がある規約（BEM など）を CLAUDE.md に冗長に書いても効果は薄く、むしろ200行制限を圧迫するだけ。

## うまく動かない時のチェック

| 症状 | 確認 |
|------|------|
| 振る舞いが変わらない | `/clear` で履歴をリセットしたか / CLAUDE.md がプロジェクトルートにあるか |
| Claude が CLAUDE.md を読んでいない | プロンプトの先頭で「CLAUDE.md は読みましたか？」と聞いて確認 |
| 色がハードコードのまま | CLAUDE.md の「ハードコード禁止」が伝わっていない → 「`var(--color-*)` を必ず使う」と書き直す |
| 新しい CSS ファイルを作られた | 「新しい CSS ファイルは作らない」が読み飛ばされた → 太字や箇条書きで強調する |
| Step 1 でも綺麗な結果が出てしまう | 既存の Button / Card に強い BEM・トークン痕跡があるため起こりがち。**色を「赤・黄・緑」と既存トークンにない色で指定**しているか確認（指示文をそのまま使う） |

## ポイント整理

- CLAUDE.md は **Claude Code 起動時に自動で読まれる**ファイル
- 置き場所: プロジェクト (`<root>/CLAUDE.md`) / ユーザー (`~/.claude/CLAUDE.md`) / ローカル (`CLAUDE.local.md`)
- 優先度: **Local > Project > User**（下が上書き）
- 書くのは: 技術スタック / コマンド / 規約 / ディレクトリ構成
- 書かないのは: 秘密情報 / 一般論 / 重複内容
- 便利機能: `@<path>` で取り込み / `/init` で雛形 / `/memory` で編集 / 自然文「これ CLAUDE.md に追加して」で追記

## 持ち帰りアイデア（自分の現場で書きたい例）

- デザイントークンの一覧と使い分けルール
- ブレイクポイント（`--breakpoint-md: 768px` などの定義）
- プロジェクトで禁止している CSS プロパティ（`float` は使わない等）
- レビュー時に気をつけてほしいこと
- よく使うスニペットや雛形の場所

**「同じ説明を2回以上した」と感じたら、それが CLAUDE.md に書くサイン**。
