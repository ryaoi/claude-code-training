# 演習3: Skills

## 目的

Slash Command（**手動で呼ぶ**）と対比して、Skill（**自動で発動する**）を作って体感する。
特に `description` の書き方で発動率がどう変わるかを観察する。

今日は **`a11y-check`（アクセシビリティをチェックする Skill）** を作る。

## 前提

- 演習2を完了している（Tag コンポーネントを追加済みの状態）

---

## 解説

### Skill とは

**「いつ使うべきかを自分で書いておく能力」**。
Claude がユーザーの依頼内容を見て、**自動で発動するかどうか判断**する。

```
ユーザー: 「この HTML、a11y 大丈夫？」
       ↓
Claude が登録済み Skill 一覧を見る
       ↓
description が一致した Skill が自動発動
       ↓
その Skill の手順に従って処理
```

### Slash Command との決定的な違い

| | Slash Command | Skill |
|---|---|---|
| 発動 | **手動**（`/name` と打つ） | **自動**（依頼の文脈に Claude がマッチ判定） |
| 例 | `/explain css/...` | 「アクセシビリティ確認して」と話しかけるだけ |
| 向いてる用途 | **自分が**決まった操作をしたい時 | Claude に**いつでも**使ってほしい能力 |

> 同じことを Slash Command でも Skill でも書けるが、
> **「自分が呼びたい」=Slash Command、「Claude に自動で使ってほしい」=Skill** で使い分ける。

### Skill の格納場所と構造

```
sample-project/
└── .claude/
    └── skills/
        └── a11y-check/        ← Skill 名のディレクトリ（必須）
            ├── SKILL.md       ← 必須。frontmatter + 本文
            ├── templates/     ← 出力テンプレート（任意）
            ├── examples/      ← 使用例（任意）
            ├── references/    ← 詳細資料（任意）
            └── scripts/       ← 補助スクリプト（任意）
```

**ポイント**: Skill は**ディレクトリ単位**。`SKILL.md` だけでも動くが、補助ファイルを同梱できる。
（Slash Command は `.md` ファイル1個だったのと対照的）

### なぜディレクトリ構造なの？

複雑な Skill ほど、本文（SKILL.md）と参照資料を分けたほうが読みやすい:

- **`SKILL.md`**: 「何をするか」「どう判断するか」のメインフロー（500行以内推奨）
- **`templates/`**: 出力フォーマットの雛形
- **`references/`**: 「色のコントラスト基準は WCAG AA で 4.5:1」みたいな詳細仕様
- **`examples/`**: 入出力サンプル

→ Claude は必要な時だけ参照ファイルを読む。**コンテキスト消費を抑えながら**機能を増やせる。

### frontmatter の主要フィールド

```yaml
---
name: a11y-check                              # 必須・小文字とハイフンのみ
description: HTML/CSS のアクセシビリティを点検する。「a11y」「アクセシビリティ」「読み上げ大丈夫？」などの依頼で使う
---
```

| フィールド | 役割 | 必須？ |
|-----------|------|--------|
| `name` | Skill 名（小文字・数字・ハイフンのみ、64文字以内） | 必須 |
| `description` | **発動条件**を書く（最大 1024 文字） | 必須 |
| `allowed-tools` | 使えるツール制限（例: `Bash(npm *)`） | 任意 |
| `model` | モデル指定（`opus`, `sonnet` など） | 任意 |

> 他にも `disable-model-invocation`（自動発動を切る）、`context: fork`（別コンテキストで動かす）など応用フィールドがあるが、今日は触らない。

### `description` が**発動条件**

ここが今日の最重要ポイント。
`description` は「何をするか」だけでなく、**「いつ呼んでほしいか」**を書く。

**悪い例**:
```yaml
description: a11y を見る
```

**良い例**:
```yaml
description: HTML や CSS のアクセシビリティ（a11y）をチェックする。ユーザーが「アクセシビリティ」「a11y」「読み上げ大丈夫？」「視覚障害」「キーボード操作」などについて確認や改善を依頼したときに使う。alt 属性・ラベル・見出し階層・コントラスト・キーボード操作可否を確認する。
```

**コツ**:
- ユーザーが**実際に使いそうな自然な日本語キーワード**を含める
- 「〜について依頼されたとき」「〜したいとき」と**発動シーンを明示**
- 「何をチェックするか」を簡潔に列挙

### 3段階のロード（コンテキスト効率）

Skill には**進捗的開示（Progressive Disclosure）**という仕組みがある:

| 段階 | いつロード | コスト |
|------|-----------|--------|
| ① メタデータ（`name` + `description`） | **常時**（起動時） | 〜100 tokens / Skill |
| ② SKILL.md 本文 | **発動時のみ** | 〜5000 tokens 以内推奨 |
| ③ 参照ファイル（templates/ などの中身） | **必要になった時だけ** | 都度 |

→ Skill が10個あっても、起動時は**メタデータ1KB**だけ。本格的に動くのはマッチした1つだけ。
**だからプロジェクトに Skill をたくさん置いても重くならない**。

### 置き場所スコープ

| 場所 | スコープ | 用途 |
|------|---------|------|
| `<project>/.claude/skills/<name>/` | プロジェクト | チームで共有したい Skill |
| `~/.claude/skills/<name>/` | 全プロジェクト | 自分の癖の Skill |
| エンタープライズ設定 | 組織全体 | 会社支給 |

優先度: **エンタープライズ > 個人 > プロジェクト**（同名の場合）。

### 組み込み Skill

Claude Code に最初から入っている Skill もある:

- `/simplify` — 変更ファイルを並列レビュー
- `/debug` — 問題のある会話のデバッグ
- `/loop` — 指定間隔でプロンプトを繰り返す

> `/` だけ打てば補完候補に出る。組み込み Skill は `/<name>` で手動でも呼べる。

### 3機能の使い分け表（今日の総まとめ）

| | Slash Command | CLAUDE.md | Skill |
|---|---------|------|---------|
| いつ動く | **手動**（`/name`） | **起動時に常時** | **自然文マッチで自動** |
| 配置 | `.claude/commands/<name>.md` | `<root>/CLAUDE.md` | `.claude/skills/<name>/SKILL.md` |
| 実体 | `.md` ファイル1個 | `.md` ファイル1個 | **ディレクトリ** |
| 用途 | 自分の定型作業 | プロジェクトの前提知識 | Claude にいつでも使ってほしい能力 |

## ハンズオン手順

### Step 1: Skill を作る

`sample-project/.claude/skills/a11y-check/SKILL.md` を新規作成:

```markdown
---
name: a11y-check
description: HTML や CSS のアクセシビリティ（a11y）をチェックする。ユーザーが「アクセシビリティ」「a11y」「読み上げ大丈夫？」「視覚障害」「キーボード操作」などについて確認や改善を依頼したときに使う。alt 属性・ラベル・見出し階層・コントラスト・キーボード操作可否を確認する。
---

以下のチェックリストで、対象の HTML/CSS をレビューしてください。
対象が指定されていない場合は `index.html` を見る。

## チェック項目

1. `<img>` に `alt` 属性があるか（装飾画像なら `alt=""`）
2. `<button>` / `<a>` / `<input>` にラベル（テキストか `aria-label`）があるか
3. 見出しレベルが飛んでいないか（h1 → h3 などはNG）
4. インタラクティブ要素にフォーカススタイル（`:focus` / `:focus-visible`）が当たっているか
5. テキストと背景のコントラスト比が WCAG AA（4.5:1）を満たしそうか（明らかに薄い色の指摘のみ）
6. `<html lang="...">` が指定されているか
7. 色だけで状態を伝えていないか

## 出力フォーマット

問題ごとに以下の形式で書く。問題が無い項目は省略してよい。

\`\`\`
### [Severity] <短い見出し>

**該当箇所**: <ファイル:行番号>
\`\`\`html
<該当 HTML を抜粋>
\`\`\`

**問題**: <何が問題か。1〜2行>

**改善案**:
\`\`\`html
<修正後の HTML 例>
\`\`\`
\`\`\`

`[Severity]` には `Critical` / `Warning` / `Info` のいずれかを入れる。
最後に「Critical: N件 / Warning: N件 / Info: N件 / 問題なし」形式のサマリを出す。
```

> `name` の `a11y-check` と、ディレクトリ名 `a11y-check/` を**一致させる**こと。

### Step 2: 自然文で発動させる

Claude Code を `/clear` してから、こう聞く:

```
この HTML、アクセシビリティ的に大丈夫？
```

**観察ポイント**:
- Claude が `a11y-check` Skill を**自動で**呼び出すか？
  - 画面に `⏺ Skill(a11y-check)` のようなツール実行行が出れば発動している
  - 呼ばれずに普通の返答だけが返ってきたら未発動（description を疑う）
- チェックリストに沿った構造で出力されるか？

### Step 3: わざと問題を作って再実行

`index.html` の Card A に、`alt` 属性なしの画像を追加する（自分で書いてもいいし、Claude に「画像を追加して」と頼んでもよい）:

```html
<article class="card">
  <h3 class="card__title">カード見出し A</h3>
  <img src="https://placehold.jp/300x150.png">  <!-- alt なし -->
  <p class="card__body">カードの本文がここに入ります。短い説明文を想定。</p>
</article>
```

ブラウザでリロードして表示を確認したあと、`/clear` してから:

```
このページの a11y を確認して
```

**観察ポイント**:
- Skill が自動発動して、追加された `<img>` の alt 不足を**ちゃんと見つける**
- 改善案に `alt="..."` の例が出る

### Step 4: description を雑にして比較

`SKILL.md` の frontmatter を一時的に書き換える:

```markdown
---
name: a11y-check
description: a11y を見る
---
```

`/clear` してから、**キーワードを含まない間接的な依頼**で聞いてみる:

```
このページを見て、操作で困る人がいないか教えて
```

**観察ポイント**:
- Skill が呼ばれず、普通の応答だけが返ってくる（`⏺ Skill(a11y-check)` の行が出ない）
- Claude は依頼文から a11y の話だと推測できないため、description が「a11y を見る」だけだとマッチしない

> **なぜ「このページの a11y を確認して」では検証にならないか**: この文には "a11y" という単語が直接入っているので、雑な description でも単語一致で発動してしまう。
> description の良し悪しは「**ユーザーが a11y という単語を使わずに聞いたとき**」にハッキリ差が出る。

確認のため、**詳細な description に戻して同じ依頼文で再実行**する:

```
このページを見て、操作で困る人がいないか教えて
```

今度は `⏺ Skill(a11y-check)` が呼ばれ、チェックリスト形式で返ってくるはず。

> 教訓: `description` には**ユーザーが実際に発しそうな自然な言い回し**（「読み上げ大丈夫？」「キーボードで操作できる？」「困る人いない？」など）を含めておく。これが**発動率を決める**。

確認後、元の詳細な description に戻す。

### Step 5: Slash Command と Skill の違いを整理

| 観点 | Slash Command (`/explain`) | Skill (`a11y-check`) |
|------|---------------------------|---------------------|
| 呼び出し | 手動（`/explain ...` と打つ） | 自動（依頼内容にマッチで発動） |
| トリガー | ユーザーが入力 | `description` のマッチング |
| 用途 | 自分が決まった操作をする時 | Claude にいつでも使ってほしい能力 |
| 配置 | `.claude/commands/name.md` | `.claude/skills/name/SKILL.md` |

---

## 期待される結果

- 詳細な `description` だと**間接的な依頼（「困る人いない？」など）でも**自動発動し、固定フォーマットで a11y 結果が出る
- 雑な `description` だと、直接キーワード（"a11y"）が入った依頼には反応しても、**間接的な依頼では発動しない**
- 「いつ呼ぶか」を `description` に書くのが Skill 設計の核心、と理解する

## うまく動かない時のチェック

| 症状 | 確認 |
|------|------|
| Skill が呼ばれない | `description` に「いつ使うか」が書かれているか / `/clear` したか / 依頼文が曖昧すぎないか |
| 雑な description なのに呼ばれる | 依頼文に "a11y" など description の単語が直接入っていないか（単語一致で発動している） |
| ファイルが認識されない | パスが `.claude/skills/<name>/SKILL.md` の形か（`<name>/` ディレクトリ必須） |
| `name` とディレクトリ名が違う | 一致させる（`a11y-check` なら両方 `a11y-check`） |
| alt 不足を見つけてくれない | 「実際に `index.html` を読んでチェック」と一言追加 |
| フォーマットがバラバラ | SKILL.md の出力フォーマット指示を太字や箇条書きでより明確に |

## ポイント整理

- Skill は **ディレクトリ + `SKILL.md`** が基本構造（Slash Command との大きな違い）
- `description` は「**いつ呼んでほしいか**」を書く。発動条件そのもの。
- Progressive Disclosure のおかげで、Skill が増えても起動コストはほぼ変わらない
- 補助ファイル（`templates/` `references/` など）を同梱できる

## 持ち帰りアイデア（HTML/CSS 現場で使える Skill 例）

- **bem-lint** — CSS から BEM 違反のクラス名を列挙
- **token-check** — ハードコードされた色や余白を検出してトークンを提案
- **responsive-review** — メディアクエリを読んで、レスポンシブで崩れそうな箇所を指摘
- **copy-polish** — HTML 内の日本語コピーを読みやすく書き換える
- **brand-voice** — 「常に使ってほしい」UI 文言ガイドライン（`description` を「ユーザー向け文言を書く時に使う」と書く）

> **設計のコツ**: 「自分なら**どう頼まれた時にこれを使ってほしい**か」を先に決め、
> その表現を `description` に詰め込むと、自動で発動してくれる。
