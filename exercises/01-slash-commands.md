# 演習1: Slash Commands

## 目的

「自分の作業手順を Markdown で定義して、`/コマンド名` で呼び出せる」体験をする。
今日は **`/explain`（コードを日本語で解説してくれるコマンド）** を作る。

## 前提

- `sample-project/` を Claude Code で開いている
- ターミナルで `claude` を起動済み
- ブラウザで `index.html` を開いて初期状態を確認済み

---

## 解説

### Slash Command とは

Claude Code のプロンプトで **`/` から始まるショートカット**のこと。
よく使う作業を1コマンドで呼び出せる。

例:
```
/clear
/explain css/components.css
/audit index.html
```

### Slash Command は4種類ある

| 種類 | 例 | 配置場所 |
|------|----|----------|
| 組み込み | `/help`, `/clear`, `/init` | （Claude Code 本体） |
| **自作（今日作る）** | `/explain`, `/audit` | `.claude/commands/<name>.md` |
| プラグイン由来 | `/some-plugin-cmd` | プラグインを入れると追加される |
| MCP 由来 | `/mcp__<server>__<name>` | MCP サーバ接続時に追加される |

### まず覚えたい組み込みコマンド（4つだけ）

| コマンド | 用途 |
|----------|------|
| `/help` | ヘルプ画面（バージョン情報、ドキュメントリンク等） |
| `/clear` | 会話履歴をリセット（**演習2・3で何度も使う**） |
| `/init` | プロジェクトを分析して `CLAUDE.md` の雛形を自動生成（演習2で再登場） |
| `/model` | 使うモデルを切り替え |

> **使えるコマンド一覧を見たい時は `/` だけ打つ** → 補完候補に全部出る（組み込み + 自作の両方）。

### 自作コマンドの仕組み

**ファイル1個 = コマンド1個**。とてもシンプル:

```
sample-project/
└── .claude/
    └── commands/
        ├── explain.md    ← /explain になる
        └── audit.md      ← /audit になる
```

**ファイル名がそのままコマンド名**。`explain.md` を作れば `/explain` が増える。

> **命名の注意**: 組み込みコマンド（`/init` `/review` `/security-review` など）と同じ名前で作ると、自作の方が優先されて組み込みが隠れる。
> 衝突を避けるには、`/` を打って候補を確認してから命名する。今日 `/review` ではなく `/audit` にしているのもこの理由。

### ファイルの中身は2パート

```markdown
---
description: 何をするコマンドか（補完候補に出る説明文）
---

ここに本文（Claude にやってほしいことを日本語で書く）
```

- 上の `---` で囲まれた部分が **frontmatter**（メタ情報）
- 下が **本文**（Claude への指示。普通の Markdown でOK）

### frontmatter で使える主なフィールド

| フィールド | 役割 | 今日使う？ |
|-----------|------|-----------|
| `description` | 補完候補に出る説明文 | 使う |
| `allowed-tools` | コマンドが使えるツールを制限（安全装置） | 今日は触らない |

> `allowed-tools` を書くと「git だけ許可する」のように制限できる。チーム配布する時に重要。今日はスキップ。

### 本文で使える3つの便利機能

#### ① `$ARGUMENTS` — 引数を受け取る

コマンド名のあとに書いた文字列が `$ARGUMENTS` に入る:

```markdown
$ARGUMENTS について解説してください。
```

実行例:
```
/explain css/components.css の .card クラス
```

→ `$ARGUMENTS` = `css/components.css の .card クラス`

#### ② `@<ファイルパス>` — ファイル内容を本文に埋め込む

```markdown
必ず @CLAUDE.md を確認してから作業してください。
```

→ コマンド実行時に `CLAUDE.md` の中身がそのまま展開される。
よく参照するファイルを毎回読み込ませたい時に便利。

#### ③ `` !`<シェルコマンド>` `` — シェル出力を本文に埋め込む

```markdown
現在のディレクトリ構成:
!`ls -la`

これを踏まえて...
```

→ コマンド実行時に `ls -la` が実行され、結果が埋め込まれる。

> 今日のハンズオンでは ① の `$ARGUMENTS` を中心に使う。② ③ は「こういうのもある」と覚えておけば OK。

### プロジェクト用 vs 個人用

| 配置 | スコープ | 用途 |
|------|---------|------|
| `<project>/.claude/commands/` | プロジェクト内のみ | チームで共有したい定型作業 |
| `~/.claude/commands/` | 全プロジェクト | 自分の癖・好み |

> 今日は **プロジェクト用** に作る。`sample-project/.claude/commands/` の下に置く。

## ハンズオン手順

### Step 1: コマンドファイルを作る

`sample-project/.claude/commands/explain.md` を新規作成し、以下を貼り付ける:

```markdown
---
description: 指定したファイルや CSS クラス・HTML 要素を、日本語で構造化して解説する
---

$ARGUMENTS について、以下のフォーマットで日本語で解説してください。

## 役割
このコードが何のためにあるか。1〜2文で。

## 使われている場所
どのファイルから参照されているか。`grep` などで実際に確認する。

## 仕組み
重要な CSS プロパティ・HTML 構造・命名規則などを、箇条書きで。
初学者にもわかる言葉で。

## 注意点・改善余地
触る時に気をつけること、改善できそうな点。
無ければ「特になし」。
```

### Step 2: コマンドを実行する

Claude Code のプロンプトで:

```
/explain css/components.css の .card クラス
```

期待される動作:
- `$ARGUMENTS` に「`css/components.css` の `.card` クラス」が入る
- Claude が `components.css` を読み、`.card` の役割・使われ場所・仕組みを日本語で構造化して返す

### Step 3: 別のものでも試す

```
/explain css/tokens.css 全体
```

```
/explain index.html の <section class="section"> の構造
```

**観察ポイント**:
- `$ARGUMENTS` に何を入れても、同じフォーマットで返ってくる
- 中身が完全に違うコードでも、コマンド1つで「同じ視点で読んでもらえる」

### Step 4: もう1つコマンドを作る（5分・余裕があれば）

`.claude/commands/audit.md` を作成（`/review` は組み込みと衝突するので `/audit` にする）:

```markdown
---
description: HTML/CSS ファイルをレビューし、日本語で改善案を出す
---

$ARGUMENTS をコードレビューしてください。
以下の観点で日本語でフィードバックしてください:

- **命名**: BEM になっているか（`.block__element--modifier`）
- **色や余白**: ハードコード（例: `#2563eb`）ではなく `var(--color-*)` などのトークンを使えているか
- **アクセシビリティ**: `alt` / `aria-*` / 見出しレベルなど基本項目
- **その他**: 重複・冗長・将来のメンテで詰まりそうな箇所

各指摘は以下の形式で書く:

\`\`\`
### 該当箇所
<ファイル:行番号 + 抜粋>

### 問題
<何が問題か。1〜2行>

### 改善案
<具体的な修正案。コード例があると良い>
\`\`\`

問題が無ければ「特に指摘事項なし」と返す。
```

実行例:

```
/audit css/components.css
```

---

## 期待される結果

- `/explain <何か>` で、好きなファイルやクラスを同じフォーマットで解説してもらえる
- `/audit <ファイル>` で、レビュー観点が固定された自動レビューが走る
- `.claude/commands/` に置いた `.md` ファイルがそのままコマンドになることを体感

## うまく動かない時のチェック

| 症状 | 確認 |
|------|------|
| `/explain` が候補に出ない | ファイルパスが `sample-project/.claude/commands/explain.md` か / Claude Code を起動し直す |
| 説明が英語になる | コマンド本文に「日本語で」が書かれているか |
| `$ARGUMENTS` がそのまま表示される | コマンド本文中の表記（小文字や全角混在）を確認 |
| 解説が雑（ファイルを読んでいない） | コマンドに「ファイルを実際に読んで」と一言追加 |

## ポイント整理

- Slash Command は **`.md` ファイル1個で1コマンド**。実体は単純なテキスト。
- frontmatter の `description` は補完候補に出る説明文
- 本文では3つの便利機能が使える: `$ARGUMENTS` / `@file` / `` !`shell` ``
- プロジェクトローカル: `.claude/commands/`、ユーザー全体: `~/.claude/commands/`
- **「自分が毎回お願いしている定型作業」をコマンドにすると一気に楽になる**

## 持ち帰りアイデア（自分の現場で作りたい例）

- `/responsive-check` — レスポンシブ崩れがないか観点を出す
- `/copy-improve` — HTML 内の日本語コピーを読みやすく書き換える
- `/component-skeleton` — 新コンポーネントの HTML/CSS 雛形を BEM で生成
- `/contrast` — 指定した2色のコントラスト比を計算して WCAG 適合か返す

「毎回同じことを Claude に頼んでるな」と気づいたら、それがコマンド化のサイン。
